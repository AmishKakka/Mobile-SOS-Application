In the HTML demo code, it does not create a random ID. I hardcoded it to const incidentId = '999'; just so it could easily open two browser windows and guarantee they would connect to the exact same room without having to type anything.

However, in  actual production app, it will absolutely be a unique room every single time.

Here is exactly how that unique ID lifecycle works in the real app:

1. The ID comes from MongoDB, not Socket.io

When the Victim presses the physical SOS button, they don't connect to Socket.io immediately.
First, app makes a standard HTTP POST request to your Node.js backend to declare an emergency. Then, backend creates a new "Incident" document in MongoDB. MongoDB automatically generates a mathematically unique ID for it (e.g., 65d3a2...1c).

2. Node.js distributes the ID

Node.js replies to the Victim's phone: "Okay, your Incident ID is 65d3a2...1c. Go join that Socket room."
When Node.js uses Firebase to wake up the nearby Helpers, it includes that exact same ID in the push notification payload: "Someone needs help! If you accept, join room 65d3a2...1c."

3. Socket.io creates the room "On Demand"

Socket.io is incredibly memory-efficient. We do not have to pre-create or configure rooms on the server.
The exact millisecond the Victim's React Native app calls socket.join('incident_65d3a2...1c'), Socket.io checks its memory.

If the room doesn't exist, Socket.io creates it instantly.

When the Helper calls socket.join('incident_65d3a2...1c'), Socket.io drops them into that existing room.

4. Automatic Destruction (Memory Cleanup)

We also don't have to worry about deleting the rooms. When the incident is marked "Resolved", both the Victim and Helper apps will close their socket connections.
